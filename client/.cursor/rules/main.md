# Project Rules (Strict)

## Стек и структура
Проект на **Next.js 16 (canary)**, **React 19**, **TypeScript** с **App Router** и корневой раскладкой **без `src/`**.

# Соглашения
- Стили: **Tailwind CSS v4**, токены и база — в `app/globals.css`.
- Импорты кода проекта: алиас `@/*` (см. `tsconfig.json`).
- Иконки: **lucide-react** (`components.json`).
- Изменения в области Next.js — сверка с `node_modules/next/dist/docs/`.

- Минимизировать сложность

- Следовать существующим паттернам проекта
- Не изобретать новые архитектуры локально

## Именование файлов

- Использовать camelCase для всех файлов и папок.
- Первое слово с маленькой буквы, каждое следующее с большой.
- Названия должны быть короткими и описательными.
- Не использовать version suffixes: `button2`, `cardNew2`, `finalHeader`.
- Не использовать общие названия: `utils.ts`, `helpers.ts`, `temp.ts`.

## Components

- Компоненты именовать по назначению:
  - `heroSection.tsx`
  - `productCard.tsx`
  - `siteHeader.tsx`

## Utility Files

- Утилиты именовать по действию:
  - `formatPrice.ts`
  - `getProducts.ts`
  - `validateEmail.ts`

## Каталоги
- `app/` — маршруты, layouts, страницы, глобальные стили (`globals.css`).
- `components/` — переиспользуеммые компоненты проекта
- `lib/` — утилиты и инфраструктурные хелперы(по необходимости).
- `public/` — статические ассеты.
- `hooks/` — клиентские хуки (алиас `@/hooks` из `components.json`; по мере появления).

## Роуты страниц
- `/` → `app/page.tsx` (главная страница).
- `/catalog` → `app/catalog/page.tsx` (страница каталога).
- `/product/[slug]` → `app/product/[slug]/page.tsx` (динамическая страница товара по slug).
- Корневая раскладка для всех роутов: `app/layout.tsx`.

## Правила создания роутов
- Каждый новый маршрут в `app/` должен иметь минимум `page.tsx`.
- Для долгих загрузок добавлять `loading.tsx` рядом с `page.tsx`.
- Для обработки ошибок сегмента добавлять `error.tsx` (клиентский компонент).
- Для пустых/несуществующих сущностей добавлять `not-found.tsx`.
- При необходимости общей структуры для сегмента добавлять `layout.tsx` в соответствующую папку маршрута.

## Правила создания секций на странице
- Любая прикладная секция страницы (`hero`, `catalog`, `contact`, `footer` и т.п.) должна быть полноширинной оболочкой: внешний блок секции — `w-full`.
- В сам элемент `<section>` не передавать прикладные пропсы; секция служит структурной оболочкой и описывается через классы.
- Внутренний контент секции (сетка, колонки, текстовые блоки, карточки) размещать через класс **`.section-layout`** из `app/globals.css` (см. также `data-and-props.mdc`).
- Адаптивность секций задавать через классы и токены из `app/globals.css` (брейкпоинты, контейнеры, отступы), без локального хардкода вне принятой системы.
- Не дублировать контейнерные отступы `px-*` и второй `container` внутри секций, если используется `.section-layout`; базовые горизонтальные отступы и ширина столбца задаются им.

## Стили, адаптив и пропсы (обязательно для всех компонентов)
- **Внешний вид и адаптив** задаются только **классами в разметке**: Tailwind с префиксами брейкпоинтов (`sm:`, `md:`, `lg:`, `xl:` и т.д.) и семантические/токенные значения из `app/globals.css`. Медиазапросы выполняет браузер.
- **Запрещено** ради отступов, размеров, сетки или типографики вводить пропы вроде `size`, `variant` (как «экран»), `breakpoint` или писать **JS** (`matchMedia`, `resize`, хуки ширины), если тот же результат выражается через responsive-классы.
- **Пропсы допустимы** для данных и поведения: контент (текст, подписи, URL), числа (счётчики), коллбеки, controlled state, композиция через `children`, проброс `className`/`style` для точечной настройки с родителя, `asChild`, атрибуты доступности.
- Проп «визуального режима по ширине», если его можно заменить классами `md:*` / `lg:*`, **не добавлять**.
- **Исключение (крайний случай):** когда без JS или без отдельного пропа **нельзя** выразить требование не про CSS (например, другая **структура узлов** в зависимости от данных с сервера, а не от viewport). Такие места кратко пояснять комментарием в коде.
